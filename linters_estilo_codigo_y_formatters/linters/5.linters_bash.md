# Linters para Shell Scripts (bash)

En iCarto usamos _bash_ como shell por defecto dado que está presente en la mayoría de sistemas y consideramos útil usar los [bashims](https://mywiki.wooledge.org/Bashism). En este documento usamos "shell", "shell script" y "bash" como sinónimos.

Hay varias opciones disponibles en distinto estado de madurez:

-   [Shellcheck](https://github.com/koalaman/shellcheck). Buena documentación de cada _smell_. Buena integración con el resto de herramientas.
-   [bashate](https://github.com/openstack-dev/bashate). Desarrollado por openstack lo llaman el pep8 de bash. Fundamentalmente cuestiones estilísticas, como que sólo se usan espacios y son múltiplos de 4. Muy ajustado al estilo openstack. Con decisiones opinativas como no usar `do` en la misma línea que `for`. Es mejor usar shellcheck y editorconfig o un _formatter_.

# Shellcheck

[Shellcheck](https://github.com/koalaman/shellcheck) es la mejor herramienta de análisis estático de código para Bash. Cada uno de los _smells_ está [bien documentado en la wiki](https://github.com/koalaman/shellcheck/wiki/Checks). No analiza cuestiones de estilo (como el indentado o fin de línea), para ello se puede usar un formatter como `shfmt` o `bashate`.

Para excluir un código de ser validado es necesario [usar una directiva](https://github.com/koalaman/shellcheck/wiki/Directive) en el propio código, u [otras opciones](https://github.com/koalaman/shellcheck/wiki/Ignore) habituales. También se pueden configurar algunas opciones a través de un [fichero de configuración](https://github.com/koalaman/shellcheck/blob/master/shellcheck.1.md#rc-files).

La versión de los repositorios suele estar desactualizada (2 años en Ubuntu 18.04). Para pre-commit se puede usar el Docker oficial. Para usarlo con el editor lo más recomendable es descargar un binario compilado y ponerlo en el PATH. Otra opción de instalación [es usar este wrapper instalable con pip](https://github.com/shellcheck-py/shellcheck-py). Que en realidad lo que hace es descargar el binario compilado y ponerlo en el path.

Tiene plugins para los principales editores ([Atom](https://atom.io/packages/linter-shellcheck), VS Code, [Emacs](http://skybert.net/emacs/bash-linting-in-emacs/), ...). La mayoría de entornos CI/CD lo soportan directamente o permiten instalarlo. No tiene integración oficial para pre-commit pero [hay PRs abiertos](https://github.com/koalaman/shellcheck/issues/1861) usando docker.

Los parámetros se pueden poner en `SHELLCHECK_OPTS` o pasar a través de la línea de comandos

-   `-x`. Por defecto shellcheck no "sigue" ficheros que se incluyan en el que se está analizando mediante `source` . `.`. Este parámetro activa la lectura pero sólo debe usarse con fuentes fiables.
-   `-x file`. En lugar de _sourcear_ todos los ficheros, sólo los indicados.
-   `-P` permite especificar directorios donde buscar los directorios a _sourcear_ en caso de que shellcheck no pueda encontrarlos. `SCRIPTDIR` es para que use el directorio del script como fuente.
-   `-a`. No sólo lintea el fichero actual si no también los _sourced_, en caso de haber usando `-x`
-   `-s shell`. Por defecto shellcheck se fija en el shebang para decidir que shell se quiere usar. Con esto se sobreescribe el comportamiento. En general es mejor no usarlo de modo que si el script no tiene shebang salte el error `SC2148`.
-   `--list-optional` nos da una lista de análisis que no están activos por defecto y que se pueden activar con `-o`

Examina todos los ficheros que se le pasan como parámetros pero no tiene una opción recursiva. [En la wiki](https://github.com/koalaman/shellcheck/wiki/Recursiveness) proveen algunos ejemplos de como hacerlo, para analizar todo un repositorio.

**Algunos problemas**

-   `-x` [no se puede especificar](https://github.com/koalaman/shellcheck/issues/1818) desde el fichero de configuración. Esto hace que haya que configurar el flag en cada herramienta (IDE, pre-commit, ...), y el fichero de configuración pierda algo de valor. Otra alternativa es usar [SHELLCHECK_OPTS](https://github.com/koalaman/shellcheck/blob/master/shellcheck.1.md#environment-variables) o hacer un wrapper.

-   No es raro encontrarse con problemas a la hora de hacer `source` de otros ficheros o intentar crear librerías de scripts. Los problemas son variados como el error relativo a `openBinaryFile`, el `SC1090`, el `SC1094` o un comportamiento distinto entre el uso desde el IDE y la línea de comandos. Hay una opción muy poco documentada `-P SCRIPTDIR` que parece incluir automáticamente en el PATH de `shellcheck` el directorio en el que se encuentra un script que está siendo analizado. Por prueba y error se ha encontrado que `-x -P SCRIPTDIR` y escribir el código como se indica en `inters_bash_shellcheck_sourced.md` funciona

# Herramientas descartadas o con poco mantenimiento

-   [bashlint](https://github.com/skudriashev/bashlint) Un herramienta sencilla escrita en python que sólo comprueba tres reglas de estilo: "Trailing whitespace", "Blank line contains whitespace", "Trailing semicolon". Es más práctico en este caso delegar directamente en un formatter como _shfmt_ o en editorconfig.

-   [Shellharden](https://github.com/anordal/shellharden) es un programa escrito en rust (hay que descargar el compilador para poder usarlo) que hace dos cosas. Avisar cuando estás usando variables en shell si entrecomillar, lo que es considerado [uno de los mayores problemas de bash](http://mywiki.wooledge.org/BashPitfalls) y tratar de arreglarlo con su opción `--transform`. No compensa para "simplemente" reformatear poniendo las comillas. Y el linting de _shellcheck_ es más potente.

-   [grunt-lint-bash](https://github.com/oxyc/grunt-lint-bash) Una tarea de grunt que ejecuta `bash -n` sobre una serie de scripts. Shellcheck da información mucho más útil que `bash -n`.

-   [checkbashisms](http://manpages.ubuntu.com/manpages/trusty/en/man1/checkbashisms.1.html) es un script incluido en los [devscripts de debian](https://packages.debian.org/sid/devscripts) que analiza un script de shell en busca de construcciones que no sean un [estándar de POSIX](http://pubs.opengroup.org/onlinepubs/009695399/utilities/xcu_chap02.html). Hay plugins de linting al menos para [Atom](https://atom.io/packages/linter-checkbashisms) y [Emacs](https://github.com/Gnouc/flycheck-checkbashisms). Probablemente con usar _shellcheck_ es suficiente, pero no lo he probado.

# Configuración iCarto

En iCarto usamos shellcheck, instalándolo como se indica a continuación:

```
VERSION="v0.8.0" # or "v0.4.7", or "latest"
url="https://github.com/koalaman/shellcheck/releases/download/${VERSION}/shellcheck-${VERSION}.linux.x86_64.tar.xz"
wget -qO- ${url} | tar -xJv
mkdir -p "${HOME}/bin
cp "shellcheck-${VERSION}.linux.x86_64/shellcheck" "${HOME}/bin/_shellcheck"
echo '#!/bin/bash

_shellcheck --norc -x -P SCRIPTDIR --enable=all --exclude=SC2312 "$@"' > "${HOME}/bin/shellcheck"
chmod u+x "${HOME}/bin/shellcheck"
rm -rf "shellcheck-${VERSION}.linux.x86_64"
shellcheck --version
```

Al usarlo mediante un wrapper no es necesario un fichero de configuración, o parámetros adicionales en el editor, pre-commit, ... Se pierde la opción de configuración específica a nivel proyecto, pero no hay solución "óptima".

Lo lanzamos mediante pre-commit:

```yaml
repos:
    - repo: local
      hooks:
          - id: shellcheck
            name: shellcheck
            entry: shellcheck
            types: [shell]
            verbose: false
```

Incluimos un script en `package.json` con como se ejecutaría para todo el repositorio: `npm run lint:bash`. Da algunos problemas al no poder ignorar ciertos ficheros desde la línea de comandos, pero no hay una solución "óptima".

```json
"scripts": {
    "lint:bash": "git ls-files -c -o --exclude-from=.ignore '**.sh' | xargs shellcheck",
},
```
